| 角色 | 描述 | 典型场景 |

| **平台管理员** | 负责 TKE 平台的规划、建设和运维 | 集群规划、资源盘点、容量管理 |

## 场景 1: 新员工集群接入

### 场景描述

**作为** 一名新入职的开发工程师  
**我想要** 快速获取公司 TKE 集群的访问权限  
**以便于** 能够在本地使用 kubectl 进行开发和调试

### 业务背景

新员工入职后，需要：

1. 了解公司有哪些 K8s 集群可用
2. 找到自己团队应该使用的集群
3. 获取集群访问凭证
4. 验证能够正常连接集群

### AI 执行步骤

```
1. 查询所有可用集群，了解集群全貌
   - 列出集群名称、版本、节点数、状态
   - 识别出运行中的集群

2. 根据集群名称或标签找到目标集群
   - 假设团队使用的集群名包含 "test" 或 "dev"

3. 检查目标集群的健康状态
   - 集群状态是否为 Running
   - 节点是否全部就绪

4. 检查集群的访问端点
   - 是否已开启外网访问（远程办公需要）
   - 如未开启，提示需要申请开通

5. 获取 kubeconfig
   - 获取外网 kubeconfig
   - 验证 kubeconfig 格式正确

6. 输出接入指南
   - kubeconfig 保存路径建议
   - kubectl 测试命令
```

### 验收标准

| AC | 描述 | 验证方法 |
|----|------|----------|
| AC1 | 能够列出所有集群并显示关键信息 | 返回集群列表，包含名称、状态、版本 |
| AC2 | 能够识别运行中的集群 | 过滤出 status=Running 的集群 |
| AC3 | 能够检查集群健康度 | 节点全部 running 视为健康 |
| AC4 | 能够判断外网访问是否可用 | 端点状态为 Created |
| AC5 | 能够获取有效的 kubeconfig | kubeconfig 包含必要配置 |
| AC6 | 生成完整的接入指南 | 输出可执行的操作步骤 |

## 场景 2: 集群资源巡检

### 场景描述

**作为** 一名 SRE 工程师  
**我想要** 对所有 TKE 集群进行快速健康巡检  
**以便于** 及时发现潜在问题，保障服务稳定性

### 业务背景

SRE 需要定期（每天/每周）对集群进行巡检：

1. 检查所有集群的运行状态
2. 检查节点健康状况
3. 检查访问端点可用性
4. 生成巡检报告，标记异常项

### AI 执行步骤

```
1. 获取所有集群列表

2. 对每个集群进行健康检查:
   a. 集群状态检查
      - Running: 正常
      - Creating/Upgrading: 警告（进行中）
      - Abnormal: 严重告警
      
   b. 节点健康检查
      - 统计 running/initializing/failed 节点数
      - 计算健康率 = running / total
      - 健康率 < 100%: 警告
      - 健康率 < 80%: 严重告警
      
   c. 端点可用性检查
      - 检查内网/外网端点状态
      - 未开启: 信息提示
      - Creating: 警告
      - Created: 正常

3. 汇总巡检结果
   - 统计正常/警告/严重告警数量
   - 列出所有异常项
   - 给出处理建议

4. 生成巡检报告
```

### 验收标准

| AC | 描述 | 验证方法 |
|----|------|----------|
| AC1 | 能够遍历所有集群 | 对每个集群都执行检查 |
| AC2 | 能够正确判断集群状态 | Running=正常，Abnormal=严重 |
| AC3 | 能够计算节点健康率 | 健康率 = running节点数 / 总节点数 |
| AC4 | 能够检查端点状态 | 识别端点是否可用 |
| AC5 | 能够生成分级告警 | 正常/警告/严重三级 |
| AC6 | 生成结构化巡检报告 | 包含汇总和详情 |

## 场景 3: 新项目环境规划

### 场景描述

**作为** 一名平台管理员  
**我想要** 为新项目规划 TKE 集群部署方案  
**以便于** 选择合适的网络、可用区和集群配置

### 业务背景

新项目立项后，需要：

1. 盘点现有基础设施资源（VPC、子网、安全组）
2. 评估可用区的容量和限制
3. 了解可选的 K8s 版本
4. 生成环境规划建议

### AI 执行步骤

```
1. 盘点 VPC 资源
   - 列出所有 VPC
   - 统计每个 VPC 的子网数量和可用 IP
   - 识别适合部署的 VPC（非默认、IP 充足）

2. 分析子网分布
   - 按可用区统计子网
   - 计算每个可用区的可用 IP 总量
   - 识别 IP 资源紧张的子网

3. 检查安全组配置
   - 列出可用安全组
   - 识别适合 TKE 的安全组（名称包含 tke/k8s/container）

4. 分析可用区状态
   - 哪些可用区可用
   - 推荐的可用区（基于子网 IP 充足度）

5. 获取 K8s 版本信息
   - 可选版本列表
   - 推荐版本（最新稳定版）

6. 生成规划建议
   - 推荐的 VPC
   - 推荐的子网和可用区
   - 推荐的 K8s 版本
   - 预估可支撑的节点规模
```

### 验收标准

| AC | 描述 | 验证方法 |
|----|------|----------|
| AC1 | 能够盘点 VPC 资源 | 列出所有 VPC 及其子网 |
| AC2 | 能够计算 IP 可用量 | 汇总每个 VPC/子网的可用 IP |
| AC3 | 能够分析可用区分布 | 按可用区统计资源 |
| AC4 | 能够推荐部署方案 | 基于资源情况给出建议 |
| AC5 | 能够推荐 K8s 版本 | 列出版本并标记推荐 |
| AC6 | 生成完整规划报告 | 包含资源盘点和建议 |

## 场景 4: 集群访问故障排查

### 场景描述

**作为** 一名开发工程师  
**我想要** 排查无法连接 TKE 集群的问题  
**以便于** 快速恢复开发工作

### 业务背景

开发人员反馈 kubectl 无法连接集群，需要：

1. 确认集群是否正常运行
2. 检查访问端点状态
3. 验证 kubeconfig 是否有效
4. 定位问题原因并给出解决方案

### AI 执行步骤

```
1. 验证集群存在且运行中
   - 查询集群状态
   - 如果不存在：提示集群 ID 错误
   - 如果非 Running：提示集群状态异常

2. 检查节点健康状况
   - 如果没有健康节点：可能影响 API Server

3. 检查访问端点
   - 内网端点状态
   - 外网端点状态
   - 根据用户网络环境判断应该使用哪个端点

4. 获取并验证 kubeconfig
   - 检查 server 地址是否与端点匹配
   - 检查证书是否存在

5. 生成诊断报告
   - 问题定位
   - 解决建议
```

### 验收标准

| AC | 描述 | 验证方法 |
|----|------|----------|
| AC1 | 能够验证集群状态 | 正确识别集群是否正常 |
| AC2 | 能够检查端点可用性 | 识别端点状态 |
| AC3 | 能够验证 kubeconfig | 检查关键配置 |
| AC4 | 能够定位问题原因 | 给出具体问题描述 |
| AC5 | 能够给出解决建议 | 针对问题给出操作步骤 |

## 场景 6: 集群扩容前检查

### 场景描述

**作为** 一名运维工程师  
**我想要** 在集群扩容前进行资源检查  
**以便于** 确保扩容操作能够成功执行

### 业务背景

集群需要扩容节点，扩容前需要：

1. 检查目标子网的 IP 是否充足
2. 检查目标可用区是否可用
3. 检查安全组配置
4. 评估扩容风险

### AI 执行步骤

```
1. 获取集群当前配置
   - 所在 VPC、子网、可用区
   - 当前节点数量

2. 检查子网 IP 余量
   - 获取子网可用 IP 数
   - 计算扩容所需 IP（节点数 + Pod 预留）
   - 判断是否充足

3. 检查可用区状态
   - 目标可用区是否可用

4. 检查安全组
   - 安全组是否存在
   - 是否有必要的规则（TKE 需要的端口）

5. 生成扩容检查报告
   - 资源是否充足
   - 风险评估
   - 扩容建议
```

### 验收标准

| AC | 描述 | 验证方法 |
|----|------|----------|
| AC1 | 能够获取集群网络配置 | 返回 VPC、子网信息 |
| AC2 | 能够检查 IP 余量 | 计算可用 IP 数 |
| AC3 | 能够检查可用区状态 | 返回可用区状态 |
| AC4 | 能够评估扩容风险 | 给出风险等级 |
| AC5 | 能够生成检查报告 | 包含检查结果和建议 |

## 场景 8: 多租户权限管理与 Agent 分发（🆕 新场景）

### 场景描述

**作为** 一名集群管理员  
**我想要** 快速为新团队创建 K8s 访问权限并生成 AI Agent 安装 Prompt

### 业务背景

企业 K8s 集群通常是多团队共享的，集群管理员面临以下挑战：

1. **权限配置繁琐**：为每个租户创建 RBAC 角色、RoleBinding、ServiceAccount 需要大量手工操作
2. **凭证分发困难**：生成 kubeconfig 后，如何安全、便捷地交付给租户是个问题
3. **知识门槛高**：即使有了凭证，租户（尤其是开发者）也不一定会用 kubectl

TKE Skill v2.0 的多租户 RBAC 管理功能，可以将这三个问题一次性解决。

### AI 执行步骤

```
1. 解析用户权限需求
   - 租户名称（如 team-frontend）
   - 目标命名空间（如 frontend, frontend-staging）
   - 权限级别（readonly/developer/admin/custom）

2. 创建 RBAC 资源
   - ServiceAccount: tke-tenant-{租户名}
   - Role: 基于权限级别模板
   - RoleBinding: 绑定 SA 和 Role

3. 获取租户 Token
   - 使用 kubectl create token
   - 支持设置过期时间

4. 生成一键安装 Prompt
   - 包含 kubeconfig 配置
   - 包含 Token
   - 包含使用说明
   - 可直接发给租户执行

5. 输出安装指南
   - 租户粘贴 Prompt 到自己的 Agent
   - 自动完成配置
   - 开始使用自然语言操作 K8s
```

### 验收标准

| AC | 描述 | 验证方法 |
|----|------|----------|
| AC1 | 能够创建完整的 RBAC 资源 | SA + Role + RoleBinding 都已创建 |
| AC2 | 权限级别正确 | readonly/developer/admin 权限符合预期 |
| AC3 | 能够获取有效 Token | Token 可用于 API 认证 |
| AC4 | 能够生成可执行的 Prompt | 租户可直接使用 |
| AC5 | 租户能够正常操作 | 权限范围内的操作都能成功 |

## 场景 9: 一句话部署应用（✅ 已验证）

### 场景描述

**作为** 一名开发工程师  
**我想要** 用自然语言将应用部署到 TKE 集群  
**以便于** 无需手写 YAML 就能完成应用上线

### 业务背景

开发者用 AI 写完代码后，想部署到 K8s，却面临：

1. 需要学习 Dockerfile 编写
2. 需要理解 Deployment、Service、ConfigMap 等概念
3. 需要配置资源限制、健康检查等最佳实践
4. 需要处理镜像构建、推送等流程

### AI 执行步骤

```
1. 分析部署需求
   - 识别应用类型（静态页面/Web 应用/后端服务）
   - 确定目标集群和命名空间

2. 设计部署方案
   - 选择合适的基础镜像
   - 设计架构（如需要处理 ARM/x86 兼容性）

3. 生成 K8s 资源
   - 编写 Deployment YAML
   - 编写 Service YAML（如需外网访问）
   - 编写 ConfigMap（如需配置文件）

4. 执行部署
   - kubectl apply 部署资源
   - 等待 Pod 就绪
   - 获取访问地址

5. 验证部署结果
   - 检查 Pod 状态
   - 测试服务可访问性

### 验收标准

| AC2 | 能够生成正确的 YAML | 资源配置符合最佳实践 |

## 场景 10: 智能运维排障（✅ 已验证）

### 场景描述

**作为** 一名开发工程师或 SRE  
**我想要** 让 AI 帮我排查 K8s 应用故障  
**以便于** 快速定位问题原因并恢复服务

### 业务背景

应用部署或运行时出现问题，传统排障流程：

1. 登录控制台查看 Pod 状态
2. 查看 Events 找线索
3. 查看容器日志
4. 搜索文档/Google
5. 不行就找 SRE...

### AI 执行步骤

```
1. 获取 Pod 状态
   - kubectl get pods
   - 识别异常 Pod（Pending/CrashLoopBackOff/Error）

2. 分析 Pod 详情
   - kubectl describe pod
   - 查看 Events 和状态条件

3. 查看容器日志
   - kubectl logs
   - 分析错误信息

4. 诊断问题根因
   - 镜像拉取失败？
   - 资源不足？
   - 配置错误？
   - 架构不兼容？

5. 给出解决方案
   - 针对具体问题提供修复步骤
   - 必要时直接执行修复
```

### 验收标准

| AC | 描述 | 验证方法 |
|----|------|----------|
| AC1 | 能够识别异常 Pod | 正确找出问题 Pod |
| AC2 | 能够分析 Pod Events | 提取关键错误信息 |
| AC3 | 能够查看容器日志 | 获取应用错误日志 |
| AC4 | 能够诊断问题根因 | 给出正确的问题原因 |
| AC5 | 能够给出解决方案 | 方案可执行且有效 |

### 预期输出示例

```
